# 渠道叙事提示词

> 用途：把每条信源渠道的「技术决策过程」翻译成产品经理和老板能听懂的语言。
> 用法：每条渠道（X / 微信公众号 / YouTube / 小宇宙 / B 站 / 聚合站）单独发一次给对应的 session，最后把 6 份输出拼到汇报文档的「工程过程」章节。

---

## 提示词正文（复制下面整段发给 session）

```
# 任务：把你这条渠道的技术选型过程，翻译给产品经理和老板看

你是【渠道名：__请在这里填，比如 X / 微信公众号 / YouTube / 小宇宙 / B站 / 聚合站】的技术实施 session。
我现在要把今天的工作汇报给产品经理和老板。他们不懂代码，但他们想知道：
「为什么这件事花了这么多时间？我们绕过了什么坑？现在它稳吗？」

请按下面 7 个字段输出一份**不超过 600 字的中文叙事**，严格遵守语言要求。

## 输出格式

### 1. 一句话定义这条渠道有多难
用一个生活类比，告诉老板「抓这条渠道」相当于现实世界里的什么动作。
（例：抓 X 像是去图书馆借 76 本书，但图书馆规定每 15 分钟只能借 50 本。）

### 2. 我们一共试过几种方案（按时间顺序列 2~4 个）
每个方案一句话：「方案名 → 为什么想试 → 试完结果」。
**不要写代码、不要写库名、不要写 API 名**。用「为什么」和「结果如何」讲。

### 3. 前面的方案为什么不行
用业务语言讲根因。
**禁止**出现：限流/限速/cookie/token/反爬/header 这些词。
**允许**：「平台一天只让我们调一次」「账号会被平台标记成机器人」「需要特定地区的网络才能拿到内容」。

### 4. 最后采用的方案是什么
继续用类比。讲清楚这个方案为什么稳——它绕开了前面什么瓶颈。

### 5. 这个方案的代价（必须写，不能省）
- 维护成本：多久要人工干预一次？
- 隐性依赖：依赖哪个账号 / 哪个网络环境 / 哪个第三方服务？
- 失效场景：什么情况下它会坏？坏了多久能修好？

### 6. 「它真的能跑」的一个证据
一句话，给出可以让老板立刻验证的东西。
例：「今天跑了一次，76 个号里 73 个有新内容，3 个号本周本来就没更新。」
**不要**贴日志、不要贴 JSON。给一个数字 + 一个结论。

### 7. 给老板的一句结论
用这个模板：
「这条渠道现在【稳 / 半稳 / 还在打磨】，每天能稳定产出 X 条原文/字幕/摘要，未来 3 个月主要风险是【一句话】。」

## 语言要求（重要）

- 全篇禁止出现：API、SDK、cookie、token、proxy、限流、header、JSON、SQLite、crawler、scraper、headless、bearer、auth。
- 如果一定要提某个工具的名字（比如 WeWe-RSS、twscrape），用「我们自建的微信文章中转服务」「专门读 X 时间线的小工具」这种描述替代。
- 数字越具体越好，但不要超过 3 个数字。

## 不要做的事

- 不要罗列你做了多少 commit
- 不要展示完整命令
- 不要解释你的代码结构
- 不要写「这个方案的优势是高并发、低耦合」这种空话

开始输出。
```

---

## 使用步骤

1. **每个 session 发一份**：X、微信公众号、YouTube、小宇宙、B 站、聚合站，各一份
2. **替换第一行的渠道名**（避免它输出别的渠道）
3. **收齐 6 份输出**后，按顺序拼到汇报文档的「工程过程」章节
4. **拼接后通读一遍**：自己改 1-2 处类比，让叙事像「你」在说话而不是 AI 在说话

## 设计原理

- **第 1 条强制类比** → 老板第一秒就有画面感
- **第 3 条禁词清单** → 卡死技术黑话的退路，逼模型用业务语言
- **第 5 条必须写代价** → PM 老板最关心的「未来要不要我兜底」，写了反而加分
- **第 6 条要数字 + 结论** → 把「我做完了」从口号变成可验证的承诺
- **第 7 条模板化结论** → 6 条渠道的结论排在一起会自动形成信号面板
